## F1 ‚Äî Add-item bottom sheet has no enter/exit motion control and no reduce-motion branch

**Severity**: MAJOR
**Lens**: 2 motion
**Rule doc**: motion-rules.md ¬ß2 (open/close asymmetry) + ¬ß4 (reduce-motion)
**Render status**: code-inferred

### What's off
The Add-item sheet (`WardrobeScreen.tsx:540-602`) is a plain RN `<Modal animationType="slide">`. The platform slide is one symmetric duration for open AND close, ignoring the house signature (OPEN `motion.duration.medium 350` + `easing.enter`; CLOSE `motion.duration.normal 250` + `easing.exit`). There is no `useReducedMotion` branch, so an OS "Reduce Motion" user still gets the full slide. The sibling sheet pattern (`ItemDetailBottomSheet.tsx:33-49`) already implements the correct asymmetric `Animated.timing` with `motion.easing.enter`/`exit` ‚Äî Wardrobe's own add-sheet doesn't follow it. Same applies to the AI-processing `<Modal animationType="fade">` (`:605`).

### Evidence
- Source: `auxi/src/screens/WardrobeScreen.tsx:540` (`animationType="slide"`), `:605` (`animationType="fade"`)
- Reference impl that does it right: `auxi/src/components/features/ItemDetailBottomSheet.tsx:33-49`
- Rule: motion-rules.md ¬ß2 (house asymmetry is a MAJOR if violated) + ¬ß4 (no `useReducedMotion` on a translate = MAJOR)

### Routing
- mobile-dev ‚Äî drive the sheet with `Animated.timing` + `motion` tokens (open 350/enter, close 250/exit) and a `useReducedMotion` cross-fade fallback, mirroring `ItemDetailBottomSheet`.

## F2 ‚Äî Grid tiles and all sheet rows have no press-feedback motion (relies on opacity dim only)

**Severity**: MAJOR
**Lens**: 2 motion + 5 states + 7 native-feel
**Rule doc**: motion-rules.md ¬ß2 (press feedback = `scale.press` 0.97 + `spring.standard`)
**Render status**: code-inferred

### What's off
Every interactive surface on Wardrobe is a `TouchableOpacity` with only `activeOpacity` (tiles `0.88` `:406`, add-method rows `0.7` `:672`, empty-CTA `0.85` `:526`, category chips `0.85` in `CategoryTabs:36`). The motion system's press-feedback contract is `scale.press` (0.97) on press-in + `spring.standard` on release. An opacity-only dim with no scale reads web-like, not iOS-native (a native cell depresses). No tile, chip, CTA, or sheet row gives the documented press affordance.

### Evidence
- Source: `WardrobeScreen.tsx:403-410` (tile), `:524-534` (empty CTA), `:670-682` (method row); `CategoryTabs.tsx:34-47` (chip)
- Rule: motion-rules.md ¬ß2 row "Press feedback" ‚Äî `scale.press` + `spring.standard`; lens 7 native-feel ("a tap target with no native press feedback feels web-like")

### Routing
- mobile-dev ‚Äî add a pressable press-scale (shared `Pressable` wrapper or `Animated` scale to `motion.scale.press` on press-in, `spring.standard` back) on tiles, chips, CTAs, and method rows. Single helper, applied across the family (DRY).

## F5 ‚Äî Tiles render off-system tier tokens + two raw rgba fills where `ds.*` tokens exist

**Severity**: MINOR (√óN ‚Äî on-system values, off-system tier)
**Lens**: 1 design-system + 4 color
**Rule doc**: design-system.md ¬ß1 (read `ds.*` first) + color-rules.md ¬ß1
**Render status**: code-inferred

### What's off
The screen is lint-clean for raw hex in its own file (`auxi-lint-tokens.sh` flags ItemDetailBottomSheet, ItemDetailScreen, Database, etc. ‚Äî NOT WardrobeScreen.tsx), but it consumes the legacy `figma*`/`uac*` alias tier throughout instead of the canonical `ds.*` layer, and inlines two raw `rgba()` fills in styles:
- `tileBadge.backgroundColor: 'rgba(18, 18, 18, 0.75)'` (`:776`) ‚Äî a duplicate of the existing token `theme.colors.figmaCardTag` (`rgba(18,18,18,0.75)`, theme.ts:23). The "New" badge correctly uses a named token; the common badge re-inlines the value.
- `sheetOverlay.backgroundColor: 'rgba(18, 18, 18, 0.4)'` (`:840`) and `tilePreparingOverlay 'rgba(255,255,255,0.6)'` (`:790`) ‚Äî scrim/overlay tints with no scrim token.
- Surfaces/text use `figmaBackground`/`figmaDetailSurface`/`figmaTextPrimary`/`uacBackgroundBase` where `ds.color.cream`/`warm100`/`slate`/`ink` exist (design-system.md ¬ß1: new code reads `ds.*` first ‚Üí MINOR each).

`rgba()` is a lint escape hatch so these don't fail the gate, but the common-badge one is a literal duplicate of a named token (DRY).

### Evidence
- Source: `WardrobeScreen.tsx:776`, `:790`, `:840`; alias usages at `:688,705,713,731,...`
- Token that already exists: `theme.colors.figmaCardTag` (theme.ts:23) == the inlined `rgba(18,18,18,0.75)`

### Routing
- mobile-dev ‚Äî swap `tileBadge` fill to `theme.colors.figmaCardTag`; migrate aliases to `ds.color.*` when next touching the file (MINOR, non-blocking). Consider a scrim token for the two overlay tints.

## F6 ‚Äî `WardrobeScreen` uses `react-native` `SafeAreaView`; its grid sibling `DatabaseScreen` uses `react-native-safe-area-context`

**Severity**: MAJOR
**Lens**: 6 cross-screen + 7 native-feel
**Rule doc**: header-footer-rules.md ¬ß3a (respect bottom safe-area inset)
**Render status**: code-inferred (import + style diff)

### What's off
`WardrobeScreen.tsx:8` imports `SafeAreaView` from `react-native`. Its nearest sibling ‚Äî the visually near-identical grid screen `DatabaseScreen.tsx:10` ‚Äî imports `SafeAreaView` from `react-native-safe-area-context`, and `ItemDetailScreen.tsx:14` uses `useSafeAreaInsets()` for both top and bottom. RN core `SafeAreaView` only pads the iOS top notch and is a no-op on Android; it does NOT give the screen a usable bottom inset. Consequences on Wardrobe:
- The item-ready snackbar overlay is pinned at a hardcoded `bottom: 24` (`:697`) with no `insets.bottom`, so on a home-indicator device it can sit too close to / behind the indicator (header-footer-rules.md ¬ß3a flags a bottom-anchored control that ignores `insets.bottom` as MAJOR).
- The add-sheet uses a hardcoded `paddingBottom: 36` (`:847`) rather than `insets.bottom + ‚Ä¶`, unlike the documented `SidebarMenu` example (`insets.bottom + 24`).
The same screen family should resolve safe-area the same way; this is a cross-screen inconsistency on top of a native-feel risk.

### Evidence
- Source: `WardrobeScreen.tsx:8` (core `SafeAreaView`), `:697` (`bottom: 24`), `:847` (`paddingBottom: 36`)
- Sibling that's correct: `DatabaseScreen.tsx:10` (`react-native-safe-area-context`), `ItemDetailScreen.tsx:243,846` (`useSafeAreaInsets`)
- Rule: header-footer-rules.md ¬ß3a

### Routing
- mobile-dev ‚Äî switch to `react-native-safe-area-context` `SafeAreaView` (match Database) and feed `insets.bottom` into the snackbar overlay `bottom` and the add-sheet `paddingBottom`.

## F7 ‚Äî No error STATE; fetch failure is a transient bottom toast that leaves an empty/loaded grid behind

**Severity**: MAJOR
**Lens**: 5 component states + journey continuity
**Rule doc**: motion-rules.md/color-rules.md state-coverage (lens 5: default/loading/empty/**error**/success)
**Render status**: code-inferred

### What's off
On a `fetchItems` failure the screen shows a 4s error Toast (`:214-220`) then renders... whatever `items` already held ‚Äî usually the empty-state ("Add your first item") because `items` is still `[]`. There is no inline error state with a Retry affordance. A user whose load failed (network) sees the *empty wardrobe* messaging telling them to add items ‚Äî which is wrong and breaks journey continuity ("Where am I? ‚Äî it looks like my wardrobe is empty, but it actually failed to load"). Lens 5 requires an understandable error state; a vanishing toast over a misleading empty-state isn't one.

### Evidence
- Source: `WardrobeScreen.tsx:211-225` (catch ‚Üí toast, no error state), `:506-536` (loading ‚Üí empty fallthrough; no error branch)
- Rule: lens 5 state coverage ‚Äî "missing error state" + journey continuity (the three questions)

### Routing
- mobile-dev ‚Äî add an inline error state (icon + message + Retry that re-calls `fetchItems`) as a third branch alongside loading/empty, so a failed load is distinguishable from an empty wardrobe.
