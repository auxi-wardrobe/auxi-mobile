## Root cause (BLOCKER)

On clean relaunch (single Metro client), the app throws a fatal RN redbox at
JS bootstrap, before any screen mounts:

```
[runtime not ready]: Invariant Violation:
TurboModuleRegistry.getEnforcing(...): 'RNGoogleSignin' could not be found.
Verify that a module by this name is registered in the native binary.
```

Stack frames (from on-screen redbox element dump):

```
configureGoogleSignIn        (com.auxi bundle :108386:15)
getEnforcing                 (:2356:27)
invariant                    (:2393:25)
loadModuleImplementation / guardedLoadModule / metroRequire ...
global                       (:266685:3)
```

The JS bundle Metro is serving references the `RNGoogleSignin` native
TurboModule (`@react-native-google-signin/google-signin`), invoked from a
`configureGoogleSignIn()` call that runs at app startup. The **installed
native binary at `com.auxi` does not contain that native module**, so
`TurboModuleRegistry.getEnforcing('RNGoogleSignin')` throws and the app
crashes before rendering. Tapping the redbox **Reload** button re-runs the
same bootstrap and re-crashes → blank white screen (status bar clock only).

This is a native/JS mismatch, not a stale-bundle-needs-reload situation. The
working assumption in the dispatch ("JS-only change, terminate+relaunch pulls
the fix, NO native rebuild") does not hold on this simulator: the current JS
(uncommitted branch `feat/au-253-home-grid-view` HEAD, or a dep it introduced)
now requires a Google Sign-In native module that the installed `auxi.app`
binary was not compiled with. Reaching Home requires a native rebuild, which
is out of scope for qa-mobile and explicitly disallowed by this dispatch.

## Root cause (BLOCKER) — new module, same class

Fresh terminate + relaunch (Step 0) produced this fatal redbox at JS bootstrap,
before any screen mounts:

```
Uncaught Error
TurboModuleRegistry.getEnforcing(...): 'RNLocalize' could not be found.
Verify that a module by this name is registered in the native binary.
```

On-screen call stack (transcribed from redbox):

```
getEnforcing                 (TurboModuleRegistry.js:41:12)
<global>                     (NativeRNLocalize.js:4:48)
<global>                     (module.native.js:3)
<global>
detectDeviceLanguage         (init.ts:61:27)
<anonymous>                  (init.ts:109:72)
```

This maps directly to source:

- `auxi/src/i18n/init.ts:36` — `import { getLocales } from 'react-native-localize';`
- `auxi/src/i18n/init.ts:61` — `const deviceLocales = getLocales();` inside `detectDeviceLanguage()`
- `auxi/src/i18n/init.ts:109` — `initialLanguage = persisted ?? detectDeviceLanguage()` during `initI18n()`

`react-native-localize@^3.7.0` is declared in `auxi/package.json`, but the
native `RNLocalize` TurboModule is **absent from the installed `com.auxi`
binary**. `getEnforcing('RNLocalize')` is triggered when `getLocales` is first
resolved (native module proxy load) and throws.

Note: `detectDeviceLanguage()` already wraps the `getLocales()` *call* in a
try/catch (init.ts:60), but the failure occurs at native-module *resolution*
during bundle eval — outside the protection of that try block — so the JS
guard does not save it. This is the same root condition as run #1's
`RNGoogleSignin`: **the installed sim binary is stale relative to the JS
dependency tree, and a JS bundle refresh cannot inject a native module.**
